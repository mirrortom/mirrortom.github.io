软件设计评论

作者: 日期: 2018/3/18

软件系统设计的想法,一直没有思考过.看下大神的见解吧..

听需求,做表单,建数据库,一个软件是这样设计的吗?也许没想过这问题,但是一直在这样干!

csdn上抄的sp1234大神的回复贴子原文(贴子原文是楼主提出软件开发使用三层架构很麻烦)

1.比如说根据你说的"有人说三层改起来方便,但我不认为,因为如果数据表加了一个字段页面上你要改好个地方很麻烦"我是可以判断出你的能力上限,这其实就暴露了问题根源.

2.真正的设计师都是从前端界面出发来进行设计,复杂系统千变万化|用户需求千变万化|工程改变千变万化,哪有用什么"增删改查"单独数据库表来千篇一律地套出程序界面的呢?这种思路已经足以说明你做过什么|喜欢做什么|在什么样的公司上班了呀.所以基本上相当于你所在的公司对技术其实没有什么要求,可能主要是搞业务的,技术人员能对付就行,老板不懂技术或者起码已经对技术人员完全放弃.

3.比如说某个项目的用户拿出20个流程的100个表单,这些可能对应与你的400个数据库表,你能不去逐一分析理解流程|表单的详细内容,而仅仅去纠结数据库表?

4.实际上数据库表是任何一个刚入门的人都会的,然后有些不求上进(但是一直混迹于程序员行业)的人工作了20年可能还是在满脑子"增删改查",因为他觉得这样已经很不容易了.实际上这样就说明,一直逃避用户千变万化的需求|逃避每一个家用户(哪怕是同一个行业)千差万别的表单和流程需求,只想用初学编程时学到的"增删改查"概念来套用出来一个通用的表单界面,而并没有学过|研究过真正的解决用户需求的应用系统工程之"道"!

5.在我们的开发中,最麻烦|最垃圾|最耗费很多倍成本的地方,其实不在于用户需求多样化,而在于一些似是而非的技术人员总是拿一些固定的"数据库字段|中间件第三方类库"之类的东西来穷于应付,总是让原本应该1周搞定的事情拖延到2个月也搞不定,其原因往往在于想的太多而实际的理念和行动力太低.

6.如果你知道开发设计任何系统都应该从用户前端需求来出发|都应该 100% 地去引导和实现用户需求的复杂实用的界面,那么你就不去先耗费巨大精力去纠结和争论无用的事情了,你就能很快学会用1周来搞项目而不是用2个月去跟用户扯皮|跟领导那里拖延时间的开发之道.

7.如果你清楚地知道软件项目架构自由度概念是多么重要,你就知道技术上该站在哪一个立场上.

8.这里有一些东西是不可能抠一些技术的字眼儿就能说清楚地,不同立场不同环境的人就会有不同的理解方式.你一定要用悟性去理解才能发现问题.

9.为什么所有的技术行业,师傅基本上都不会傻到免费带徒弟?因为社会现实是残酷的.绝大多数人,学会了别人用10年研究出来的技术路线之后,他只会觉得是那几条语句或者几个工具才牛b,他不知道真正感激"人"|而失去现学现卖地去把学到的东西贱卖,急于达到自私的目的.所以所有的技术的论坛都是分层的,所有的工程师都是只对跟自己同一个圈子里的人才会说实话.能告诉你一两点关键的理念,很不容易!

10.比如说一个项目有20种订单,那么我们就会快速地用2天时间立刻产生这20种订单的数据库表,生成时间是2秒钟,而写下这20种订单的 Model 模型的代码的时间是2天时间.而这20种 Model 的代码来自于最初的用户表单的 Word 文档,上面详细说明了没一个字段的录入|计算|关联情况.用户需要什么就实现什么,此时绝不会去纠结什么"通用的订单的数据库表",此时绝不会去满脑子去想我的数据库表怎么设计才最省力.此时全部精力都在想用户的这20种看上去完全不同|千差万别|表中套表|复杂业务关联的订单的个性化的业务需求.然后直接快速地写出 Model,最后用 2|3 秒钟快速产生数据库表.

11.数据库表不是设计软件的出发点,设计实现用户交互需求界面前端应用,从来不扯什么数据库表的羁绊性问题.

12.实际上为什么有的人把大量时间放在数据库表那里?我告诉你说吧,他们基本上 99.99% 的人都不擅长前端开发,也不擅长互联网思维的产品开发,技术都是在找工作|搞局域网OA或者进销存软件的那个阶段,所以没有进入设计师的圈子.所以用你的目标来驱动你的技术转型,而不要用技术来决定自己的爱好.技术永远都是次要的.

13.好的网络软件系统给用户的印象是"操作方法流畅得酷毙了|响应速度快得没朋友",简单清新好用并且100%地符合用户传统纸质界面的设计风格,用户大屏幕展示系统的界面风格,地图系统的界面,等等风格,而不是弄什么 Excel 电子表|SQL Server 查询分析器之类的录入界面之类的 UI 设计.

14.你只要注意一点即可:任何创业都不是开发千篇一律的前端设计,都是要拿几百个新界面创新来说事儿,都需要这个能力.找到这方面的需求的技术解决方法,才能获得自由.
